## TC 到底是什么

TC 是给普通创业者使用的 **AI 创业解题教练**。它最重要的能力不是知道多少理论，而是把一团乱麻变成一个真正的问题和一个马上能验证的动作。

用小学生也能听懂的话说：

**你不知道自己该干什么，就把事情全部告诉 TC。它先帮你找到真正卡住的地方，再告诉你下一步只做哪一件事。**

比如你说：

> 我有流量、社群和几个合作项目，但最近不知道发什么，也不知道该做哪个业务。

TC 不会马上塞给你十个选题、五条赛道和一篇长报告。它会先判断：你表面上在问“发什么”，真正需要解决的可能是“未来 90 天，到底让内容为哪个业务服务”。

问题说清楚后，TC 才会给一个推荐方案、你要放弃什么、现在第一步做什么，以及用什么结果判断这一步有没有用。

它固定做四步：

```text
讲清问题 → 找到最早断点 → 做一次低成本验证 → 根据结果继续调整
```

TC 不是创业百科、暴利项目推荐器、保证赚钱的商业导师、什么问题都回答的通用知识库，也不是自动生成一堆空泛建议的聊天机器人。知识库只是它的书架，真正的产品是判断能力。

TC 由作者 [天策](https://x.com/Leobai825) 创建，`TC` 就是“天策”的名字缩写。

## 它能帮你做什么

| 你现在的情况 | TC 会帮你拿到什么 |
| --- | --- |
| 想法很多，不知道先做哪个 | 一个真正需要解决的问题和明确取舍 |
| 有流量，却没有稳定收入 | 找到从流量到咨询、报价、付款中最早断掉的一环 |
| 有项目，但一直不敢开始 | 一个 10 分钟能启动、24 小时能接触市场的动作 |
| 不知道该发什么内容 | 先判断内容要服务什么目标，再给可直接发布的文案 |
| 合作关系说不清楚 | 拆清职责、钱、客户归属、验收和退出边界 |
| 做过一次，但不知道哪里错了 | 找到最早的错误信号和下一次低成本验证办法 |

TC 不保证你赚钱，也不替你做人生决定。它做的是：**减少空想，让判断尽快接受真实市场检验。**

长期主线不是收集越来越多创业理论，而是建立一套普通人也能使用、能够持续接受市场检验的创业解题系统。

## 30 秒开始

安装后，在一个新对话里输入：

```text
/tc
```

TC 会回复：

```text
在。把事情直接发过来，乱一点也没关系。我先帮你把问题重新说清楚。
```

然后把真实情况直接发过去。你不用先整理，也不用知道该选哪个子 Skill。

也可以把问题和命令一起发：

```text
/tc 我做了一个创业社群，有流量也有收入，但不知道下一步该放大什么。
```

## 一次完整对话长什么样

下面是演示，不是编造的成功案例。

**你：**

```text
/tc 我有五万粉丝、一个付费社群，还在接几个 AI 项目。
最近什么都想做，又觉得什么都没做，不知道主线是什么。
```

**TC 第一轮：**

```text
我先不急着给方案。

我重构后的问题：
你表面上在问“该做哪个项目”，真正卡住你的是没有确定未来 90 天
让流量、社群和交付共同服务哪一个结果。

问题定义（草案）：
在不放弃现有现金流的前提下，为未来 90 天选出一条唯一主线，
并用真实回款和可重复交付判断它是否值得继续。

我只确认一个点：
这 90 天你更想优先得到稳定现金流，还是验证一个能长期放大的产品？
```

**你：**

```text
先要稳定现金流。
```

**TC 第二轮：**

```text
问题定义完成：
未来 90 天先用现有流量和社群做出可重复的稳定现金流。

我的判断：
现在不该再开新赛道，应该先把已经有人付费的需求做成一个边界清楚的产品。

推荐方案：
从最近 30 天已经成交的需求里，只选一类最容易重复交付的服务，
做成固定对象、固定结果、固定范围、固定价格的一页报价。

你要接受的代价：
暂时放弃“什么机会都能接”。

现在第一步：
今天整理最近 10 次付费或咨询记录，找出重复出现最多的问题。

有效标准：
7 天内至少向 5 个同类客户正式报价，并拿到付款或明确拒绝理由。
```

更多场景见 [真实使用示例](docs/真实示例.md)。

## TC 怎样工作

```text
真实情况
   ↓
重构问题：分清事实、解释、情绪和假设
   ↓
定义问题：说清对象、目标、限制和验证标准
   ↓
用户确认或纠正
   ↓
一个推荐方案 + 必须接受的代价
   ↓
一个最小动作 + 一个有效标准
   ↓
带真实结果回来，继续下一步
```

简单、明确的问题会直接回答；复杂问题才会先重构。TC 不会为了显得深刻，把一句简单请求强行分析成人生课题。

如果用户从零开始、没有资源或没人找他帮忙，TC 不会把“想项目”重新推回用户。它最多确认生存边界和一个真实工作/项目场景，然后主动给出一个 **项目假设**。有联网工具时先查当前对标、用户问题、替代方案、价格和规则；没有联网工具时明确缺少现实证据，并要求在24小时内找对标、访谈或报价。项目假设不是市场结论，更不是收益承诺。

完整设计见 [TC 工作原理](docs/工作原理.md)。

## 安装

### Codex、Claude Code 与其他支持 Agent Skills 的工具

在终端执行：

```bash
npx -y skills add Leobai03/tc -g --all
```

安装完成后，新开一个对话并输入 `/tc`。

### Claude Code 插件市场

```bash
claude plugin marketplace add Leobai03/tc
claude plugin install tc@tc-skills
```

### 手动下载

只有不能使用安装命令时，才打开 [GitHub Releases](https://github.com/Leobai03/tc/releases)。普通用户只下载：

- `tc-suite-x.y.z.zip`：完整套件，包含全部 7 个 Skill 和使用说明。

只想安装主入口的高级用户可以下载 `tc.zip`。不要逐个下载一排子 Skill 压缩包；仓库仍会在构建时生成它们用于测试，但正式 Release 只展示完整套件和主入口。

### 豆包、DeepSeek、Kimi 与普通聊天 AI

这些工具不一定支持原生 Skill 安装。打开 [TC 统一入口](https://leobai825.icu/tc.html)，复制轻量版提示词，作为新对话的第一条消息发送即可。

完整步骤见 [新手入门说明书](docs/新手入门.md)。

## 7 个 Skill 分别干什么

普通用户只需要记住 `/tc`。其余入口是给明确知道自己需要什么的人使用的。

| 调用入口 | 只负责什么 | 适合什么时候用 |
| --- | --- | --- |
| `/tc` | 判断现在真正该解决什么，并推进下一步 | 不知道用哪个时永远用它 |
| `/tc-diagnosis` | 重构和定义问题 | 叙述很乱、方向很多、合作或业务卡住 |
| `/tc-copy` | 生成推动当前业务环节的文案 | 要写推文、报价、私聊、招募或跟进消息 |
| `/tc-action` | 把判断变成真实市场动作 | 已经想明白，却迟迟没有行动 |
| `/tc-state` | 把当前判断、下一步和结果存到本机，或恢复上次进度 | 想跨对话继续同一个创业问题 |
| `/tc-knowledge` | 搜索 TC 公开知识与可选外部理论库，接入并检查自己的知识库 | 查历史原文、DBS Markdown、方法出处、飞书或本地资料 |
| `/tc-update` | 从官方仓库更新 TC | 想检查并安装最新版 |

当前只发布 7 个边界清楚的 Skill。`tc-knowledge` 有独立的搜索脚本和资料治理任务，不是为了凑数量创建的空插件。

## 仓库里的文件夹都是干什么的

```text
tc/
├── skills/                    # TC 真正执行任务的 7 个 Skill
│   ├── tc/                    # 主入口：判断当前最该处理什么
│   │   ├── SKILL.md           # 主流程，相当于 TC 的总说明书
│   │   ├── agents/            # 在 Codex 等工具里显示的名称与简介
│   │   ├── assets/            # 普通聊天 AI 可复制的轻量版
│   │   └── references/        # 按问题读取的核心参考源、方法与 6 个专项知识包
│   ├── tc-diagnosis/          # 问题重构与定义
│   ├── tc-copy/               # 商业文案
│   ├── tc-action/             # 真实市场行动
│   ├── tc-state/              # 本地保存、恢复和汇总创业进度
│   ├── tc-knowledge/          # 搜索内置知识，并按许可同步、检索 DBS Markdown
│   └── tc-update/             # 官方更新
├── 知识库/                    # 核心参考源，以及知识从哪里来、怎样验证
│   ├── 核心参考源/            # 技术合伙人与天策的两套核心判断框架
│   ├── Skill知识包/           # 按具体问题装配的 6 本薄教材
│   ├── 原子库/                # 22 条带出处与边界的知识卡片
│   ├── 公开内容索引/          # 489 条历史公开原创内容
│   └── 外部理论库/            # 第三方来源卡；正文从原作者仓库按需同步
├── docs/                      # 给人看的新手教程、原理、示例和版本说明
├── tests/                     # 防止 TC 越改越虚的行为验收场景
├── tools/                     # 校验、构建和发版脚本
├── .codex-plugin/             # Codex 插件信息
├── .claude-plugin/            # Claude Code 插件市场信息
└── VERSION                    # 当前唯一版本号
```

一句话理解：

- `skills/` 是会干活的人；
- `references/` 是干活时才翻的教材；
- `知识库/` 是教材怎么来的；
- `tests/` 是考试卷；
- `tools/` 是打包和检查机器；
- `docs/` 是给使用者看的说明书。

## 知识库里有什么

TC 使用分层、按需读取的参考资料，不会每次全部塞给 AI：

- 问题重构、创业链条、生存边界；
- 内容到收入、X 起号、报价与成交；
- 市场验证、行动推进、收入记分；
- 合作边界、失败复盘、公开与合规；
- 外部资料提炼、用户知识库接入、反馈闭环；
- 489 条截至 2026-07-14 的天策公开原创内容索引；
- 2 份核心参考源：技术合伙人的商业与 AI 实操框架、天策的公开实践与创业价值观；
- 22 条带日期、链接、证据等级和适用边界的知识原子；
- 6 个覆盖诊断、行动、内容、产品、组织合作和 AI 落地的知识包；
- 1 个可选 DBS 外部理论库入口：对话态从原作者仓库同步 Markdown 并搜索；维护态把3至5条相关记录加工成不含正文的本机 L1 研究候选；
- 1 份统一营收、回款、毛利、验证等口径的高频概念词典；
- DBS 协同、跨平台安装与轻量版分发。

入口：[知识库导航](知识库/README.md) · [核心参考源](知识库/核心参考源) · [搜索 Skill](skills/tc-knowledge/SKILL.md) · [TC 参考资料目录](skills/tc/references) · [来源与验证规则](知识库/来源与验证.md)

原推索引只负责“天策过去公开说过什么”，默认创业解题不会检索。它不代表 TC 认可每一句，不能覆盖今天的业务状态，也不能用作者经历替用户生成项目或推荐方向。核心参考源、原子和知识包只提供经过提炼的通用方法，并服从用户当前事实和真实市场证据。

DBS 推文集采用 CC BY-NC 4.0，因此全文不会放进 TC 的 Apache-2.0 安装包。用户确认仅作署名的非商业研究后，`tc-knowledge` 才会直接从原作者仓库同步 Markdown 到本机。维护态候选也固定为本机、不可商用、不可自动升级；取得独立市场证据后仍需人工许可审查。商业文案、课程、付费交付和产品不得复制或改写原文，除非另行取得作者许可。接入说明见 [DBS 外部理论库来源卡](知识库/外部理论库/dbs-books.md)。

TC 不把观点伪装成事实。动态平台规则、法律、投资、医疗、价格与业务数据必须在使用时重新核对。

## 它不做什么

- 不把自己包装成创业百科或赚钱项目库；
- 不推荐所谓暴利项目，也不保证赚钱；
- 不承诺“用了就能赚钱”；
- 不编造案例、收入、客户评价或权威背书；
- 不替用户自动发送邮件、发布内容或联系别人；
- 不自动收集、上传或发送用户的全部对话；
- 不帮助欺骗消费者、规避平台审核或承诺收益；
- 不用一堆框架掩盖一个本来很简单的问题。

## 与 DBS 的关系

TC 可以与 [dontbesilent 的 dbskill](https://github.com/dontbesilent2025/dbskill)协同，但不是其官方组成部分，也不复制或重新分发 DBS 的 Skill 正文。

两者的简单分工是：DBS 有更丰富的专项工具；TC 更强调先找到当前真正的问题，再用一个方案和真实市场动作把事情往前推。
