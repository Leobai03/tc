## TC 到底是什么

用小学生也能听懂的话说：

**你脑子里有一团乱线，TC 先帮你找到线头，再告诉你先拉哪一根。**

比如你说：

> 我有流量、社群和几个合作项目，但最近不知道发什么，也不知道该做哪个业务。

TC 不会马上塞给你十个选题、五条赛道和一篇长报告。它会先判断：你表面上在问“发什么”，真正需要解决的可能是“未来 90 天，到底让内容为哪个业务服务”。

问题说清楚后，TC 才会给一个推荐方案、你要放弃什么、现在第一步做什么，以及用什么结果判断这一步有没有用。

TC 由作者 [天策](https://x.com/Leobai825) 创建，`TC` 就是“天策”的名字缩写。

## 它能帮你做什么

| 你现在的情况 | TC 会帮你拿到什么 |
| --- | --- |
| 想法很多，不知道先做哪个 | 一个真正需要解决的问题和明确取舍 |
| 有流量，却没有稳定收入 | 找到从流量到咨询、报价、付款中最早断掉的一环 |
| 有项目，但一直不敢开始 | 一个 10 分钟能启动、24 小时能接触市场的动作 |
| 不知道该发什么内容 | 先判断内容要服务什么目标，再给可直接发布的文案 |
| 合作关系说不清楚 | 拆清职责、钱、客户归属、验收和退出边界 |
| 做过一次，但不知道哪里错了 | 找到最早的错误信号和下一次低成本验证办法 |

TC 不保证你赚钱，也不替你做人生决定。它做的是：**减少空想，让判断尽快接受真实市场检验。**

## 30 秒开始

安装后，在一个新对话里输入：

```text
/tc
```

TC 会回复：

```text
在。把事情直接发过来，乱一点也没关系。我先帮你把问题重新说清楚。
```

然后把真实情况直接发过去。你不用先整理，也不用知道该选哪个子 Skill。

也可以把问题和命令一起发：

```text
/tc 我做了一个创业社群，有流量也有收入，但不知道下一步该放大什么。
```

## 一次完整对话长什么样

下面是演示，不是编造的成功案例。

**你：**

```text
/tc 我有五万粉丝、一个付费社群，还在接几个 AI 项目。
最近什么都想做，又觉得什么都没做，不知道主线是什么。
```

**TC 第一轮：**

```text
我先不急着给方案。

我重构后的问题：
你表面上在问“该做哪个项目”，真正卡住你的是没有确定未来 90 天
让流量、社群和交付共同服务哪一个结果。

问题定义（草案）：
在不放弃现有现金流的前提下，为未来 90 天选出一条唯一主线，
并用真实回款和可重复交付判断它是否值得继续。

我只确认一个点：
这 90 天你更想优先得到稳定现金流，还是验证一个能长期放大的产品？
```

**你：**

```text
先要稳定现金流。
```

**TC 第二轮：**

```text
问题定义完成：
未来 90 天先用现有流量和社群做出可重复的稳定现金流。

我的判断：
现在不该再开新赛道，应该先把已经有人付费的需求做成一个边界清楚的产品。

推荐方案：
从最近 30 天已经成交的需求里，只选一类最容易重复交付的服务，
做成固定对象、固定结果、固定范围、固定价格的一页报价。

你要接受的代价：
暂时放弃“什么机会都能接”。

现在第一步：
今天整理最近 10 次付费或咨询记录，找出重复出现最多的问题。

有效标准：
7 天内至少向 5 个同类客户正式报价，并拿到付款或明确拒绝理由。
```

更多场景见 [真实使用示例](docs/真实示例.md)。

## TC 怎样工作

```text
真实情况
   ↓
重构问题：分清事实、解释、情绪和假设
   ↓
定义问题：说清对象、目标、限制和验证标准
   ↓
用户确认或纠正
   ↓
一个推荐方案 + 必须接受的代价
   ↓
一个最小动作 + 一个有效标准
   ↓
带真实结果回来，继续下一步
```

简单、明确的问题会直接回答；复杂问题才会先重构。TC 不会为了显得深刻，把一句简单请求强行分析成人生课题。

完整设计见 [TC 工作原理](docs/工作原理.md)。

## 安装

### Codex、Claude Code 与其他支持 Agent Skills 的工具

在终端执行：

```bash
npx -y skills add Leobai03/tc -g --all
```

安装完成后，新开一个对话并输入 `/tc`。

### Claude Code 插件市场

```bash
claude plugin marketplace add Leobai03/tc
claude plugin install tc@tc-skills
```

### 手动下载

打开 [GitHub Releases](https://github.com/Leobai03/tc/releases)，下载：

- `tc.zip`：只安装最主要的 `/tc`；
- `tc-suite-x.y.z.zip`：完整套件，包含全部 6 个 Skill 和使用说明；
- 其他单独 ZIP：只在你明确需要某个子 Skill 时下载。

### 豆包、DeepSeek、Kimi 与普通聊天 AI

这些工具不一定支持原生 Skill 安装。打开 [TC 统一入口](https://leobai825.icu/tc.html)，复制轻量版提示词，作为新对话的第一条消息发送即可。

完整步骤见 [新手入门说明书](docs/新手入门.md)。

## 6 个 Skill 分别干什么

普通用户只需要记住 `/tc`。其余入口是给明确知道自己需要什么的人使用的。

| 调用入口 | 只负责什么 | 适合什么时候用 |
| --- | --- | --- |
| `/tc` | 判断现在真正该解决什么，并推进下一步 | 不知道用哪个时永远用它 |
| `/tc-diagnosis` | 重构和定义问题 | 叙述很乱、方向很多、合作或业务卡住 |
| `/tc-copy` | 生成推动当前业务环节的文案 | 要写推文、报价、私聊、招募或跟进消息 |
| `/tc-action` | 把判断变成真实市场动作 | 已经想明白，却迟迟没有行动 |
| `/tc-state` | 把当前判断、下一步和结果存到本机，或恢复上次进度 | 想跨对话继续同一个创业问题 |
| `/tc-update` | 从官方仓库更新 TC | 想检查并安装最新版 |

当前只发布 6 个边界清楚的 Skill。不会为了看起来厉害，制造一堆没有独立用途的空插件。

## 仓库里的文件夹都是干什么的

```text
tc/
├── skills/                    # TC 真正执行任务的 6 个 Skill
│   ├── tc/                    # 主入口：判断当前最该处理什么
│   │   ├── SKILL.md           # 主流程，相当于 TC 的总说明书
│   │   ├── agents/            # 在 Codex 等工具里显示的名称与简介
│   │   ├── assets/            # 普通聊天 AI 可复制的轻量版
│   │   └── references/        # 17 份按问题读取的方法、边界与公开知识包
│   ├── tc-diagnosis/          # 问题重构与定义
│   ├── tc-copy/               # 商业文案
│   ├── tc-action/             # 真实市场行动
│   ├── tc-state/              # 本地保存、恢复和汇总创业进度
│   └── tc-update/             # 官方更新
├── 知识库/                    # 知识从哪里来、怎样验证、怎样新增案例
├── docs/                      # 给人看的新手教程、原理、示例和版本说明
├── tests/                     # 防止 TC 越改越虚的行为验收场景
├── tools/                     # 校验、构建和发版脚本
├── .codex-plugin/             # Codex 插件信息
├── .claude-plugin/            # Claude Code 插件市场信息
└── VERSION                    # 当前唯一版本号
```

一句话理解：

- `skills/` 是会干活的人；
- `references/` 是干活时才翻的教材；
- `知识库/` 是教材怎么来的；
- `tests/` 是考试卷；
- `tools/` 是打包和检查机器；
- `docs/` 是给使用者看的说明书。

## 知识库里有什么

TC 主入口内置 17 份按需读取的参考资料，不会每次全部塞给 AI：

- 问题重构、创业链条、生存边界；
- 内容到收入、X 起号、报价与成交；
- 市场验证、行动推进、收入记分；
- 合作边界、失败复盘、公开与合规；
- 外部资料提炼、用户知识库接入、反馈闭环；
- 12 条带日期和公开链接的天策历史知识原子；
- DBS 协同、跨平台安装与轻量版分发。

入口：[知识库导航](知识库/README.md) · [TC 参考资料目录](skills/tc/references) · [来源与验证规则](知识库/来源与验证.md)

TC 不把观点伪装成事实。动态平台规则、法律、投资、医疗、价格与业务数据必须在使用时重新核对。

## 它不做什么

- 不承诺“用了就能赚钱”；
- 不编造案例、收入、客户评价或权威背书；
- 不替用户自动发送邮件、发布内容或联系别人；
- 不自动收集、上传或发送用户的全部对话；
- 不帮助欺骗消费者、规避平台审核或承诺收益；
- 不用一堆框架掩盖一个本来很简单的问题。

查看当前版本：打开 [VERSION](VERSION)。

```text
/tc-update
```

或重新执行：


每个正式版本的“改了什么、为什么改、怎样验证”都会写在 [GitHub Releases](https://github.com/Leobai03/tc/releases)，不会再只留一条空 Changelog。

## 与 DBS 的关系

TC 可以与 [dontbesilent 的 dbskill](https://github.com/dontbesilent2025/dbskill)协同，但不是其官方组成部分，也不复制或重新分发 DBS 的 Skill 正文。

两者的简单分工是：DBS 有更丰富的专项工具；TC 更强调先找到当前真正的问题，再用一个方案和真实市场动作把事情往前推。
