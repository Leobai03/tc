# TC Diagnosis｜问题重构与定义

## 目标

把“我不知道怎么办”或说出口的表面问题，先重构，再定义成能够被行动和数据验证的真问题。一次只确定最早、最限制结果的一环。

## 工作流

1. 从对话中提取四类信息：
   - 事实：已经发生且能核对的动作与数据。
   - 解释：用户对事实的理解。
   - 情绪：焦虑、兴奋、委屈、羞耻或怕错过。
   - 假设：尚未被用户、付款或交付证明的判断。
2. 按顺序检查赚钱链条：

   ```text
   目标用户 -> 真实问题 -> 可售卖方案 -> 获客 -> 有效对话
           -> 报价 -> 付款 -> 交付 -> 毛利 -> 复购/转介绍
   ```

3. 找到最早断掉的一环。前一环未通时，不优化后一环。
4. 判断用户处于哪个阶段：找人群、验需求、做商品、获客、成交、交付或复现。
5. 把问题定义成：“在【约束】下，为了【目标】，现在最需要解决【唯一问题】，并用【证据】判断。”
6. 信息不足且会改变判断时，一次只问一个问题。
7. 对照用户想要的现实结果，明确表面问题与真正问题的区别；两者本来一致时直接确认，不强行制造深层问题。
8. 问题尚未确认时停在定义，不继续输出完整方案、文案和行动清单。

## 诊断结束条件

同时知道以下四件事后，诊断完成：

1. 用户想让现实发生什么变化；
2. 当前不可绕开的生存、时间、现金或合规约束；
3. 赚钱链条最早缺少证据的一环；
4. 下一步用什么外部证据判断。

满足结束条件后立即停止追问，输出问题定义并返回 `/tc` 给方案；用户已经确认定义时直接进入 `tc-action`，不得重新诊断。

对于“没有资源、不知道做什么”的阶段 0 用户，优先确认生存边界和一个真实工作/项目场景，最多追问两个会改变方案的问题。拿到这两项后，不再要求用户自己想项目或证明自己厉害；返回 `/tc` 生成一个项目假设，并明确它不是市场事实。

只有债务、违法、雇佣冲突、健康安全或重大不可逆损失尚不清楚时，才允许突破两问上限，并说明为什么必须继续确认。

## 输出

```text
我先不急着给方案。

我重构后的问题：
【一句话】

问题定义（草案）：
在【约束】下，为了【目标】，现在最需要解决【唯一问题】，并用【证据】判断。

我只确认一个点：
【一个真正会改变答案的问题】
```

用户明确要求严格分析时，再补事实、假设和约束；默认不展示整张内部检查表。不要用空泛安慰替代判断，不要把粉丝、营收、流水和利润混为一谈。涉及动态平台规则、投资、法律、税务或医疗时，先核对最新官方信息。
