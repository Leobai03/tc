# TC 公开知识包

这份知识包由[公开原子库](https://github.com/Leobai03/tc/blob/main/%E7%9F%A5%E8%AF%86%E5%BA%93/%E5%8E%9F%E5%AD%90%E5%BA%93/atoms.jsonl)提炼而来，只在任务确实需要天策的公开方法或历史案例时读取。它不是天策当前业务状态，也不是“照做必赚”的答案。

## 行动与验证

- `TC-ACT-001`：把目标变成“看到一个可做动作 → 做 → 记录结果 → 再做”。必须同时设置投入上限和停止条件。
- `TC-EXP-001`：第一次市场实验至少记录触达、有效咨询、订单、转化、定制需求和交付成本。来源中的数字只是 2026-02-22 的历史自述，不能作为收益承诺。
- `TC-AI-001`：用自然语言说清具体产物，再用连续反馈迭代 AI 原型；做出原型不等于需求成立。

## 用户、商品与商业模式

- `TC-PROD-001`：技术能力要变成可重复出售的产品，需要说清服务谁、解决什么问题、交付什么结果。
- `TC-NEED-001`：先问产品能否让具体用户省时间、少走弯路、降低成本或解决真实麻烦，最终以用户原话和付款验证。
- `TC-MODEL-001`：研究公开市场里的获客、收费和交付方式，再设计自己的低成本验证；不得侵权、截流或规避规则。
- `TC-USER-001`：商业拆解先回答谁会买、为什么现在买、结果被包装成什么商品。

## 内容、连接与个人 IP

- `TC-FDB-001`：公开表达可以加快外部反馈，但反馈的价值是纠正判断，不是只收获赞美。
- `TC-NET-001`：有效连接建立在互补价值或新增信息上，不是单方面索取，也不能变成操控。
- `TC-ID-001`：成长型个人 IP 可以记录做过什么、结果如何和哪里失败，不必长期扮演“什么都懂”的导师。
- `TC-CONTENT-001`：不知道发什么时，先做一件真实小事，再写问题、过程、结果和下一步。

## 合作与交付

- `TC-DELIVERY-001`：商业信任来自边界清楚、说到做到、出现问题愿意修正；承担责任不等于接受无限责任。

## 使用顺序

1. 先用当前用户的事实定义问题；
2. 只选与当前断点有关的一至三张卡片；
3. 把卡片改成待验证假设或行动，不把它当权威结论；
4. 用户带回新证据后，证据优先于卡片；
5. 引用历史数据时写清原日期，并回到原始公开链接核对。
