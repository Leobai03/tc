# DBS 协同路由

## 定位

DBS 是独立的诊断与思考工具箱，TC 是“重构问题、定义问题、给出方案”的渐进式创业解题与执行层。

不要复制、改写或内嵌 DBS 的受许可内容。把 DBS 作为用户独立安装的可选依赖，按 Skill 名称调用；没有安装时使用 TC 自己的流程。

## 单步协同规则

1. 先判断当前最靠前的卡点。
2. 只有确实需要专项诊断时，选择一个 DBS Skill。
3. 完成该 Skill 的完整流程。
4. 提取一个核心结论，不把整份报告再复述一遍。
5. 回到 TC 当前阶段：问题定义尚未确认就先确认；已经确认才给方案；方案确定后才按需生成文案或市场动作。
6. 用户带回结果后，再决定下一步，禁止预设固定长链。

## 路由地图

| 当前信号 | 优先 DBS Skill | 回到 TC 后做什么 |
| --- | --- | --- |
| 想创业、想赚钱，但目标是空话 | `dbs-goal` | 改成具体用户、问题和可检查结果 |
| 用户的问题本身说不清 | `dbs-good-question` | 形成可验证的问题说明书 |
| 表面问题、情绪和真实矛盾混在一起 | `dbs-deconstruct` | 区分事实、解释、情绪和假设，找到最早业务卡点 |
| 不确定商业模式是否成立 | `dbs-diagnosis` | 选一个最危险假设做市场验证 |
| 不知道该模仿谁 | `dbs-benchmark` | 提炼五个对标的用户、商品和成交证据 |
| 知道该做但反复拖延 | `dbs-action` | 把动作缩到十分钟并立即开始 |
| 选题和内容方向不清 | `dbs-content` | 把内容接到目标用户和商业下一步 |
| 开头留不住人 | `dbs-hook` | 选一个开头，发布真实测试 |
| 文稿怕没共鸣 | `dbs-resonate` | 保留一个核心冲突并补真实证据 |
| 逐字稿逻辑容易断 | `dbs-script-flow` | 完成可发布稿并设置转化目标 |
| 小红书标题 | `dbs-xhs-title` | 发布并记录目标用户反馈 |
| 文稿 AI 味重 | `dbs-ai-check` | 补回本人经历、原话和判断 |
| 想解释内容为什么传播 | `dbs-spread` | 把传播情绪转成目标用户洞察 |
| 大量旧内容需要资产化 | `dbs-content-system` | 建立可重复选题和案例生产线 |
| 重大选择需要长期验证 | `dbs-decision` | 设置商业假设、指标和回填日期 |
| 要保存阶段结果 | `dbs-save` | 保存本轮市场证据与下一步 |
| 要汇总多轮结果 | `dbs-report` | 生成面向团队或合伙人的业务报告 |

## 标准交接格式

每次 DBS 返回后，用以下五行继续：

```text
DBS 核心结论：
TC 当前阶段：重构 / 定义 / 方案 / 执行 / 复盘
本轮只推进：
```

只有进入方案或执行阶段，才补充市场动作和收入证据。收入证据从弱到强：目标用户原话、有效对话、明确报价、定金、回款、正毛利、复购或转介绍。

## 自媒体协同示例

用户说“我想做一个创业账号，但不知道发什么”：

1. 当前问题是内容方向，使用 `dbs-content`，不先跑全部工具。
2. 得到选题判断后，TC 先确认用户真正要解决的是表达、获客还是商业承接，再补充目标用户、这条内容承接什么商品、行动邀请是什么。
3. 用户发布后带回：曝光、主页访问、目标用户评论、私信、预约和成交。
4. 下一轮根据真实数据决定使用 `dbs-hook`、`dbs-resonate` 或直接继续同类选题。

## 创业协同示例

用户说“我想做一个 AI 简历产品”：

1. 当前问题是需求是否成立，使用 `dbs-diagnosis` 或 `dbs-benchmark` 中更必要的一个。
2. TC 把结论改成一天内的验证：五个对标、十个用户触达、三个报价。
3. 没付款时不开发完整产品。
4. 有三次相近成交后，再考虑自动化和规模化。

## 边界

- 不声称本 Skill 是 DBS 官方组成部分或获得 dontbesilent 背书。
- 对 DBS 的名称、作者和许可证保持清楚归属。
- 商业使用 TC 时，不打包或重新授权 DBS 的受限内容。
- 用户明确调用 DBS 时，尊重 DBS 的单步动态路由规则。
