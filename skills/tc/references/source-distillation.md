# 外部实操资料提炼

当用户提供某个人的大量帖子、会议记录、课程稿、聊天记录或行业资料，并要求“总结进知识库”“吸收进 TC”时，使用本文件。

目标不是复述原文，而是把资料变成可追溯、可验证、可执行、可以继续升级或降级的知识原子。

## 一、先做来源卡

每个来源先登记：

- 来源名称与链接；
- 作者或资料类型；
- 获取日期与原文最后更新时间；
- 覆盖范围；
- 是否允许公开；
- 当前定位：事实真源 / 一手经验 / 外部观点 / 灵感素材。

没有公开授权时，只能在内部知识库保存链接、摘要和判断；公开 Skill 只能保留经过改写的通用方法，不复制原文、身份、客户、收入或商业机密。

有公开授权也要逐条遵守许可。DBS 开源推文集采用 CC BY-NC 4.0，因此只能作为署名的非商业外部理论库按需检索；全文不进入 Apache-2.0 的 TC 安装包，商业输出不复制或改写原文。具体接入规则见 [DBS 开源推文集来源卡](external-sources/dbs-books.md)。

DBS 不只可以在用户点名时查询，也可以在 **TC 维护阶段** 作为上游研究材料。两种状态必须分开：

- 对话态只返回带署名和许可边界的外部观点；
- 维护态把 3 至 5 条相关记录加工成本机 L1 候选，用于提出独立假设、行动、指标和反例；
- 候选不自动进入公开 Skill，也不能仅凭“换一种说法”升级；
- 只有拿到独立市场证据并通过人工许可审查，才允许另行写成 TC 原子。

## 二、先分类，再吸收

把每条候选内容放进以下一类：

| 类型 | 含义 | 默认处理 |
| --- | --- | --- |
| 可核验事实 | 有订单、合同、后台、交付记录、用户原话或可靠公开来源 | 保留日期和口径后使用 |
| 一手经验 | 作者亲自做过，并能说明动作与结果 | 作为经验，不自动写成普遍规律 |
| 观点或假设 | 有启发，但缺少充分证据 | 放入待验证区，设计小实验 |
| 动态判断 | 涉及工具、模型、平台、价格、政策和行业状态 | 使用前重新核对 |
| 不可采用 | 违法违规、欺骗、操控、侵犯隐私、鼓励剥削或无法承受的高风险做法 | 明确剔除，不转化为行动建议 |

表达很有力量，不等于证据很强。作者很成功，也不等于每句话都正确。

## 三、知识原子格式

每条知识原子只表达一个能指导判断或行动的意思：

```markdown
## 原子标题

- 类型：QST / CON / OPI / CAS / SOL
- 来源：
- 日期：
- 证据等级：A / B / C / D
- 当前状态：候选 / 已验证 / 已失效 / 有冲突
- 是否允许公开：是 / 否

### 原始洞察

### TC 翻译
把观点改写成一句能执行、能被证伪的规则。

### 适用场景

### 最小动作

### 判断有效的证据

### 边界与停止条件

### 与其他原子的关系
回应 / 解释 / 证明 / 冲突
```

候选原子还必须包含：

- `scenario`：在哪种具体处境下使用；
- `action`：下一项最小市场动作；
- `success_metric`：用什么外部证据判断；
- `counterexample`：什么事实会推翻它；
- `license` 与 `commercial_eligible`：能否进入公开或商业安装包。

内容单元的类型沿用 DBS 内容结构化系统的五类对象：

- `QST`：值得反复追问的问题；
- `CON`：需要稳定定义的概念；
- `OPI`：有明确适用边界的观点；
- `CAS`：含场景、动作和结果的案例；
- `SOL`：可以照着执行并验收的方案。

不要把每段话都拆成原子。只有会改变后续判断、文案或行动的内容才值得进入。

## 四、四级升降级

```text
L0 原始资料
  ↓ 已标注来源、日期和权限
L1 候选原子
  ↓ 有明确场景、动作和可证伪条件
L2 已验证模式
  ↓ 多次复现，或有更强证据支持
L3 TC 稳定方法
  ↓ 多条方法长期形成一致的提问与判断方式
核心参考源
```

- `L0` 只负责留证，不代表认可。
- `L1` 可以启发实验，不能包装成定律。
- `L2` 可以在相似场景中推荐，但必须说明边界。
- `L3` 才能进入 `SKILL.md` 主流程；它必须跨案例稳定、符合公开与合规边界，并能导向真实行动。
- “核心参考源”不是更高级的事实证据，而是一套高频使用的世界观。它必须同时列出保留原则、反例、不可采用内容和冲突处理规则。

对于含 CC BY-NC 或其他受限第三方来源的 L1 候选，默认始终保留在本机研究区。L2 只代表“方法得到验证”，不自动改变版权或商业许可；许可审查和证据审查是两道独立关口。

出现反例、环境变化、证据冲突或明显副作用时，允许从 `L3` 降回 `L2/L1`，不要为了维护作者正确而维护旧结论。

## 五、从技术合伙人商业与 AI 实操资料中保留的核心框架

这批资料是 TC 在商业化、需求、产品、流量、AI 与 To B 交付问题上的主要参考来源。以下原则可以高频用于形成问题和判断，但每次仍要结合用户场景与真实证据：

1. **AI 是杠杆，不是需求。** 先确认谁为什么付钱，再判断 AI 能让哪一步更快、更便宜或更稳定。
2. **好答案从好问题开始。** 先找最影响结果、最便宜就能验证的未知数。
3. **先验证，再开发。** 先访谈、预售、发报价或手工交付；没有付款证据时，不急着做完整系统。
4. **Prompt 是任务交代。** 说清背景、目标、输入、边界、输出格式和验收，比堆神秘术语更重要。
5. **Agent 是岗位，不是魔法。** 给它职责、可用工具、状态、反馈和失败时的处理规则。
6. **用日志和验收说问题。** 技术排错要提供复现步骤、输入、输出、错误日志与期望结果，少说“它坏了”。
7. **先简化流程。** 把业务画成流程图，删除不创造价值的节点，直到再删一步会破坏交付。
8. **企业项目先划边界。** 购买者、使用者、数据来源、交付范围、验收、售后、责任和退出机制必须写清楚。
9. **内容要连接真实业务。** 分享具体方法、过程和证据；流量不能代替产品、成交和交付。
10. **重复有效动作。** 在单位经济成立后重复同一条有效链路，不因短期情绪不停换方向。

完整公开提炼见 [技术合伙人｜商业世界观与 AI 实操框架](core-sources/技术合伙人_商业世界观与AI实操.md)。私有原文与动态经营细节不进入安装包。

## 六、必须剔除或降级的内容

以下内容即使传播性强，也不得进入 TC 稳定方法：

- “一定赚钱”“降维打击”“印钞机”“所有公司都怎样”等绝对化结论；
- 未给来源的收入、客户数、成功率、市场规模和名人案例；
- 把高流量、高价格或有人购买当成产品真实有效的证明；
- 鼓励欺骗、白嫖、偷学、压榨、PUA、规避审核或把他人当工具；
- 把用户受损一概归因于“认知不足”，免除销售者的真实披露与交付责任；
- 把政治、历史、宗教或游戏隐喻直接当成商业证据；
- 对某个模型、工具、平台或技术路线的永久性判断。

## 七、提炼后的输出

默认只交付四部分：

1. 来源卡与本次处理边界；
2. 最值得保留的 5 至 10 条知识原子；
3. 待验证和明确剔除项；
4. 哪些内容进入内部知识库、哪些升级到公开 Skill，以及下一次验证动作。

用户要求“总结进知识库”时，优先更新内部知识页；只有稳定、脱敏、通用的方法才更新公开 TC。不要用一份漂亮总结冒充已经完成知识工程。
