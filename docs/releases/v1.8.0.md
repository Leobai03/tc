# TC v1.8.0｜从零项目假设与知识反馈闭环

## 这次解决什么

v1.8.0 修复两个核心问题：从零用户回答完现实情况后，TC 仍可能连续诊断，把“想项目”重新推回用户；DBS 虽然已经可搜索，却还没有进入一条可追溯、可验证、受许可约束的知识加工流水线。

同时收紧产品定位：TC 是给普通创业者使用的 AI 创业解题教练，不是创业百科或赚钱项目库。知识库只是书架，真正的产品是判断能力。

固定循环是：

```text
讲清问题 → 找到最早断点 → 做一次低成本验证 → 根据结果继续调整
```

## 七项优化

1. **产品定位护栏**
   - 主 Skill、公开介绍、轻量版和插件元数据统一为“AI 创业解题教练”；
   - 明确不推荐暴利项目、不保证收益、不输出空泛建议；
   - 新增自动化契约，防止后续迭代重新漂移成创业百科或项目库。

2. **作者经历隔离**
   - 默认知识搜索只读取核心参考源、知识包和知识原子；
   - 历史原推只在用户明确查询作者过去表达时单独返回；
   - 作者经历不得参与普通用户的项目生成、方向推荐或能力判断。

3. **DBS 研究候选**
   - 新增 `dbs-candidate`，每次只取3至5条相关记录；
   - 只保存 ID、日期、链接、主题、标签和许可证，不复制正文；
   - 候选固定保存在本机，状态为 L1、不可商用、不可自动升级；
   - 独立市场证据与人工许可审查是两道不同关口。

4. **从零项目假设**
   - 新增 `zero-to-one.md`；
   - TC 最多确认生存边界和一个真实工作/项目场景；
   - 信息足够后由 TC 主动给一个项目假设，不要求用户自己想项目；
   - 有联网工具时核对当前对标、用户问题、替代方案、价格和规则；没有联网工具时明确未知，并把第一步改成现实取证。

5. **诊断硬切换**
   - 明确诊断完成的四个条件；
   - 阶段0默认最多追问两个关键问题；
   - 满足条件后必须停止追问并切到方案或 `tc-action`。

6. **主入口瘦身**
   - 主 `SKILL.md` 从373行缩减为不到240行；
   - 主文件只保留状态、路由、核心约束和输出；
   - 详细规则按需放入从零、市场、内容、知识和边界参考。

7. **市场证据反馈**
   - `tc-state` 同时保存结构化本机状态；
   - 用户单独授权后可用 `export-evidence` 导出脱敏候选；
   - `tc-knowledge candidate-add` 只把它放入本机 L1 候选区；
   - 单个案例不能升级稳定方法，不自动上传或进入公开仓库。

## 验证

新增从零创业、联网与无联网边界、诊断切换、DBS 候选溯源、非商业许可、状态证据单独授权与脱敏的自动化测试和验收场景。
