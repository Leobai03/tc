# TC v1.4.0｜第一次真正拥有“创业存档”

发布日期：2026-07-21

## 这次解决什么

以前的 TC 能帮用户把问题说清楚、给出方案和行动，但新开对话后容易重新从头讲。知识库也主要是规则和模板，缺少能追到原文的真实知识卡片。

v1.4.0 不靠继续堆提示词解决这件事，而是新增一个可运行的状态模块、一批公开知识原子和对应自动测试。

## 用户能直接感受到的变化

- 新增 `/tc-state`：在本机保存、恢复、列出和汇总创业进度；
- `/tc` 能识别“新手、新任务、继续旧任务”三种状态；
- 新开对话后可以输入 `/tc 恢复“项目名”`，从上次下一步继续；
- 存档默认位于 `~/.tc/`，不会自动上传给作者；
- 历史公开方法会标注原推链接、日期、可信度和适用边界。

## 仓库里的真实变化

- `skills/tc-state/scripts/tc_state.py`：新增标准库 Python 脚本；
- `tests/test_tc_state.py`：自动测试保存、列表、恢复、报告和错误输入；
- `知识库/原子库/atoms.jsonl`：首批 12 条公开知识原子；
- `skills/tc/references/public-knowledge-atoms.md`：可随主 Skill 安装的按需知识包；
- `docs/DBS架构深度调研.md`：逐层解释 DBS 的根目录、29 个 Skill、知识库、工具和发布链路；
- 构建器会排除 `__pycache__` 和 `.pyc`，避免安装包混入无用缓存；
- CI 现在会运行状态脚本测试，完整性检查也会核对知识原子字段和知识包引用。

## 为什么没有照抄 DBS 的 29 个 Skill

TC 保持一个 `/tc` 主入口和五个专项 Skill。保存、恢复、报告被合并进 `tc-state`，因为它们服务同一个用户目的；拆成三个文件夹只会增加记忆和路由成本。

这次学习的是 DBS 的分层、状态闭环和可执行工具，不学习“文件夹越多越专业”。

## 验证

```bash
python3 tools/check.py

```
