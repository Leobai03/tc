# TC 工作原理

这份文档回答三个问题：TC 为什么不是一段普通提示词、仓库为什么有这些文件夹、它怎样从混乱叙述走到可执行方案。

## 一句话原理

```text
先找真问题，再给一个方案，最后用真实结果继续修正。
```

普通 AI 容易直接回答用户嘴上问的那句话。TC 会先判断“用户想问的”和“用户真正需要解决的”是不是同一件事。

- 如果本来就是同一件事，直接回答；
- 如果问题混乱或存在关键误判，先重构和定义；
- 问题确认后，才给推荐方案；
- 方案必须落到市场动作和可观察证据。

## 主流程

```text
用户的真实叙述
      ↓
区分：事实 / 解释 / 情绪 / 假设
      ↓
检查：赚钱链条最早断在哪里
      ↓
写成：有对象、目标、约束、边界、验证标准的问题
      ↓
用户确认或纠正
      ↓
一个方案 + 一个主要代价
      ↓
一个最小动作 + 一个有效标准
      ↓
用新证据更新下一步
```

赚钱链条按这个顺序检查：

```text
目标用户 → 真实问题 → 可售卖方案 → 获客 → 有效对话
        → 报价 → 付款 → 交付 → 毛利 → 复购/转介绍
```

TC 只优先解决最早断掉的一环。例如还没有任何人愿意付款时，先验证需求，不急着研究自动化和团队扩张。

## 七个 Skill 怎样配合

`/tc` 是总入口。它读取当前对话，判断此刻只需要做哪一步。

```text
                         ┌─ 问题没说清 ─→ tc-diagnosis
用户 ─→ /tc 主入口 ─────┼─ 需要对外表达 → tc-copy
                         ├─ 知道但没行动 → tc-action
                         ├─ 要保存或恢复 → tc-state
                         ├─ 要查或维护知识 → tc-knowledge
                         └─ 需要升级版本 → tc-update
```

这不是固定流水线。TC 不会默认把六个子 Skill 全部跑一遍，而是根据上一步真实结论只选当前一个。

主入口先区分三种状态：第一次使用、新任务、继续旧任务。继续旧任务时，`tc-state` 会从本机存档恢复上次的问题、决定、待验证假设和下一步，不要求用户重新讲完整背景。

## 文件夹分工

### `skills/`：真正执行任务

每个 Skill 都有一个 `SKILL.md`。它告诉 AI：什么时候触发、按什么步骤工作、哪些事情不能做、怎样输出。

主入口位于 `skills/tc/`：

```text
skills/tc/
├── SKILL.md           # 核心工作流和路由规则
├── agents/openai.yaml # AI 工具里的展示名称、简介和默认提示
├── assets/tc-lite.txt # 普通聊天 AI 使用的轻量版
└── references/        # 只在相关问题出现时读取的方法、路由与知识包
```

### `references/`：按需翻阅的教材

TC 不会把全部方法每次都塞进上下文。遇到对应任务才读取对应文件：

- 内容问题读取 `content-to-revenue.md`；
- X 起号读取 `x-growth.md`；
- 合作和市场验证读取 `market-loop.md`；
- 外部资料整理读取 `source-distillation.md`；
- 反馈问题读取 `feedback.md`；
- 安装分发读取 `distribution.md`。

这样能减少上下文浪费，也能避免不相关的方法干扰当前判断。

### `知识库/`：知识进入 TC 的规则

这里不是把所有文章堆在一起，而是说明：

- 来源怎样登记；
- 哪些是事实，哪些只是观点；
- 动态信息什么时候必须重新核对；
- 案例怎样脱敏；
- 什么内容可以进入公开 Skill，什么只能留在私有资料里。

`知识库/公开内容索引/` 保存 489 条历史原创内容，`知识库/原子库/` 保存 22 条有日期、证据等级和公开链接的知识卡片，`知识库/Skill知识包/` 保存 6 个按问题装配的专项教材。原推只能证明作者当时的公开表达，不能自动代表今天的业务状态，也不代表 TC 认可每一句。

### `tc-knowledge`：真正面向知识的脚本

`skills/tc-knowledge/scripts/tc_knowledge.py` 可以搜索原推、原子和知识包，也能输出数量、日期范围和校验结果。它与 `tc-state` 分工明确：前者管理资料依据，后者管理某个创业任务的进度。

### `tc-state`：真正面向任务的脚本

`skills/tc-state/scripts/tc_state.py` 不只是说明文字。它可以：

- 保存一份结构化创业状态；
- 列出本机项目；
- 恢复最近一次存档；
- 把多次存档合成阶段报告。

默认只写入 `~/.tc/`，不自动上传。敏感内容保存前必须提醒用户明文风险。

### `docs/`：给人看的说明书

`SKILL.md` 是给 AI 执行的；`docs/` 是给使用者和维护者理解的。两者不能混为一谈。

### `tests/`：行为考试卷

测试不只检查文件能不能打开，还检查 TC 会不会：

- 把简单问题强行复杂化；
- 一上来倾倒长报告；
- 编造数据或收益承诺；
- 把内容流量当成真实收入；
- 忽略合作、平台与隐私边界。

### `tools/`：构建和发版机器

- `check.py`：检查版本、目录、元数据、链接和必要文档；
- `build.py`：生成单 Skill 安装包和完整套件；
- `release.py`：同步所有位置的版本号。

## 三层加载方式

TC 使用 Agent Skills 常见的渐进式加载：

1. **元数据**：AI 平时只看到名称和一句用途，用来判断是否触发；
2. **SKILL.md**：触发后才读取主流程；
3. **references / assets**：只有任务确实需要时才读取。

这就是为什么 TC 不是把所有内容写进一个超长文件，也不是把一小段提示词包装成“系统”。

## 知识如何进入 TC

外部帖子、课程、飞书文档或用户历史资料不会被整包视为真理。处理顺序是：

```text
登记来源和日期
   ↓
区分事实、经验、观点、动态判断和不可采用内容
   ↓
提取会改变判断或行动的方法
   ↓
脱敏、验证、标注适用边界
   ↓
进入私有知识库或公开参考资料
```

历史内容只能证明“当时说过”，不能自动证明“现在仍然正确”。

## 隐私与反馈

TC 不会自动把用户对话发给作者，也不能暗示作者能看到使用记录。

只有用户明确同意时，AI 才能帮助整理脱敏反馈。任何反馈都应删除客户姓名、联系方式、商业机密、账户凭据和其他敏感信息。

## 怎样判断 TC 有没有用

不能只问“回答看起来聪不聪明”。更重要的是：

- 问题是否比原来更清楚；
- 用户是否接受了一个真实取舍；
- 24 小时内是否接触真实市场；
- 是否拿到用户原话、有效对话、报价、付款、毛利或复购证据；
- 新证据是否改变了下一步判断。

如果 TC 只让用户感觉舒服，却没有让现实多发生一点变化，就不算完成任务。
