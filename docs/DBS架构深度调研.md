# DBS 架构深度调研：每个文件夹到底在干嘛

> 调研对象：[`dontbesilent2025/dbskill`](https://github.com/dontbesilent2025/dbskill)
>
> 调研版本：`2.18.0`，提交 `daca0716446c255ba19ede90950450ddd2908595`
>
> 调研日期：2026-07-21
> 说明：本文研究的是 DBS 的公开仓库结构。我们学习它的工程方法，不复制其受许可证保护的内容。

## 先说人话版结论

DBS 像一所已经开起来的学校：

- `README` 是校门口的招生说明；
- `skills/` 是不同科目的老师；
- `知识库/` 是老师备课用的教材和卡片；
- `tools/` 是印试卷、检查版本和打包的机器；
- `.github/workflows/` 是自动值日生，每次更新都检查有没有出错；
- `docs/` 是新生手册、地图和演示图；
- `site/` 是学校官网；
- `.claude-plugin/` 是把老师摆进 Claude 商店的货架清单；
- `scripts/` 主要负责录宣传演示，不是给创业诊断直接使用的业务程序。

DBS 强的地方不是“文件多”，而是这些部分能接起来：入口先判断问题，专项 Skill 负责一件事，复杂 Skill 会带自己的脚本和模板，知识库提供材料，状态 Skill 让下次接着做，自动检查保证发出去的包没有明显错误。

## 根目录逐个解释

| 文件或文件夹 | 小学生版解释 | 实际作用 | TC 怎么学 |
| --- | --- | --- | --- |
| `.claude-plugin/` | Claude 商店里的货架标签 | `marketplace.json` 登记每个插件的名字、版本、分类、简介和对应 Skill 路径 | 保留，并确保每个 TC Skill 都真的登记了 |
| `.github/workflows/` | 永远不偷懒的自动检查员 | CI 检查版本与清单；Pages 发布网站；打标签后构建 ZIP 并发 Release | 保留；继续增加行为和数据检查，不只检查文件存在 |
| `docs/` | 给人看的说明书和地图 | 新手教程、Skill 关系图、知识来源图、社群入口等 | 保留；TC 增加本报告和真实演示，避免只写口号 |
| `scripts/` | 录宣传片的小摄影棚 | 用 VHS 录制安装过程并压缩 `demo.gif` | 暂不照搬。TC 没有稳定演示前，不为了好看先造摄影棚 |
| `site/` | 一张简单官网 | GitHub Pages 的静态落地页 | TC 已有个人网站承接，仓库内暂不重复造站 |
| `skills/` | 真正干活的老师们 | 29 个 Skill；大部分是说明书，少数还带脚本、模板和脚手架 | TC 保持少而清楚，但给重任务补真实脚本和验收 |
| `tools/` | 工厂里的机器 | 校验版本、检查升级合同、构建单 Skill ZIP 和全量安装包 | TC 已有基础版；增加知识原子与状态脚本检查 |
| `知识库/` | 老师背后的教材库 | 原子卡片、季度拆分、专项知识包、高频概念词典 | TC 不冒充大知识库；先做少量、有来源、能核对的公开原子 |
| `README*.md` | 不同语言的招生简章 | 解释产品、安装、用法、Skill 清单 | TC 先把中文说明做好，真实有海外用户后再翻译 |
| `VERSION` | 全校统一的版本号码 | 所有清单、徽章和发布都跟它同步 | TC 已经采用，继续保持唯一真源 |
| `LICENSE` | 哪些东西可以拿、怎么拿的规则 | DBS 使用 CC BY-NC 4.0 | TC 只学架构，不复制 DBS 文案或知识内容 |
| `demo.gif` | 30 秒演示视频 | 展示安装和调用 | 等 TC 流程稳定后再录，不把动画当核心能力 |

## `skills/` 里面的 29 个文件夹

普通用户不需要记这些名字。下面只是把每位“老师”负责什么讲清楚。

### 1. 总入口和更新

| Skill | 小学生版解释 |
| --- | --- |
| `dbs` | 前台老师。先听你说什么，再把你带到最合适的一位老师那里 |
| `dbs-update` | 软件更新员。只负责检查和安装新版 DBS |

DBS 主入口有三种工作状态：第一次来的新手引导、刚拿到新任务时的分流、做完一步后的下一步导航。它每次只选一位当前老师，不把 20 多位老师同时叫进教室。

### 2. 商业判断和思考

| Skill | 小学生版解释 |
| --- | --- |
| `dbs-diagnosis` | 给生意做体检，找出最先坏掉的一环 |
| `dbs-benchmark` | 找几个真正值得学的同类，再看人家为什么行 |
| `dbs-action` | 专治“我知道但我没做”，把事情变成下一步动作 |
| `dbs-slowisfast` | 帮你判断哪些事急着做反而更慢 |
| `dbs-deconstruct` | 把一个模糊的大词拆成能看懂的小块 |
| `dbs-goal` | 把“我想变好”改成一个清楚、能检查的目标 |
| `dbs-good-question` | 把问不清的问题改成更值得回答的问题 |

### 3. 内容生产和传播

| Skill | 小学生版解释 |
| --- | --- |
| `dbs-content` | 找选题、找角度、改内容，让文章对目标读者有用 |
| `dbs-hook` | 专门修短视频开头，减少观众第一秒就划走 |
| `dbs-xhs-title` | 给小红书内容生成和筛选标题 |
| `dbs-ai-check` | 检查一段文字是不是有很重的 AI 腔 |
| `dbs-wechat-html` | 把 Markdown 文章变成公众号能粘贴的漂亮 HTML |
| `dbs-spread` | 从传播心理看内容为什么有人转、为什么没人转 |
| `dbs-resonate` | 检查文章有没有让目标读者觉得“你在说我” |
| `dbs-script-flow` | 检查短视频逐字稿在哪里会断、会让人听不下去 |
| `dbs-content-system` | 给大量内容建仓库：拆卡片、去重复、连关系、按主题组装 |

其中 `dbs-content-system` 是 DBS 最像“完整产品”的 Skill。它不只有一份提示词，还带：

- `docs/`：验收标准和快速上手；
- `scaffold/`：一键创建内容仓库所需的目录与规则；
- `templates/`：问题、概念、观点、案例、方案等卡片模板；
- `tools/`：登记来源、抽取样本、生成草稿、找重复、连双链、组装主题等脚本。

这告诉我们：重任务不能只靠一句“请认真分析”，还要有固定输入、固定工具、固定产物和固定验收。

### 4. 对话、学习和专家讨论

| Skill | 小学生版解释 |
| --- | --- |
| `dbs-chatroom` | 选几位适合当前问题的“虚拟专家”一起讨论 |
| `dbs-chatroom-austrian` | 固定用奥地利经济学视角讨论问题 |
| `dbs-learning` | 把一门知识变成边学边问、边练边检查的课程 |

### 5. 保存进度和长期决策

| Skill | 小学生版解释 |
| --- | --- |
| `dbs-save` | 把这次讨论的重点存成一张存档卡 |
| `dbs-restore` | 下次回来时把最近的存档卡拿出来 |
| `dbs-report` | 把多张存档卡合成一份阶段报告 |
| `dbs-decision` | 给一个长期决策建档案柜，分清事实、规律、快照和未解决问题 |

这一组是 TC 当前最大的缺口。没有它，用户每次回来都要重新讲一遍，Skill 也不知道上次说的动作到底做没做。

DBS 默认把存档放在本机 `~/.dbs/`，不会因为“能存”就自动上传。这个边界值得保留：保存必须让用户知道，敏感信息默认不进入公开仓库。

### 6. Skill 基础设施和知识维护

| Skill | 小学生版解释 |
| --- | --- |
| `dbs-knowledge` | 从作者公开内容里检索相关知识卡片 |
| `dbs-agent-migration` | 把一个普通项目整理成 Agent 可以长期接手的工作台 |
| `dbs-bridge` | 给 Claude、Codex、Agents 和 Grok 之间搭兼容桥 |
| `dbs-skill-cleaner` | 给外部 Skill 做安全体检，发现危险后先报告，再让用户决定是否隔离 |

`dbs-bridge` 的脚本会把同一份 Skill 链接到不同 Agent 的技能目录，并避免直接覆盖真实文件夹。`dbs-skill-cleaner` 只读最关键的说明书和可执行脚本，查找指令劫持、敏感信息外传和危险网络命令；它先出报告，得到用户同意后才隔离。

## `知识库/` 到底是什么

DBS 的知识库不是一个大 Word 文件，而是两层：

1. **原子库**：一条公开内容拆成一张小卡片，记录原文、链接、日期、主题、适合哪个 Skill、内容类型和可信度。
2. **Skill 知识包**：把与某个任务有关的卡片重新整理成一份可读方法包。

调研版本中，原子库公开标注 4,176 条卡片，并按季度分文件。构建安装包时，脚本不会把整个知识库塞进每个 Skill；只有某个 Skill 明确引用的知识包才会一起打包。这就是“需要哪本书才拿哪本书”。

TC 应该学它的字段和按需读取方式，但不能学它的宣传数字。TC 目前只有少量公开来源，所以要诚实写“12 条可核对原子”，以后再随着真实使用增长。

## `tools/` 和自动发布到底做什么

DBS 的 `tools/` 有四类机器：

- 构建机：给每个 Skill 单独打 ZIP，再生成完整安装包；
- 版本检查机：确保 `VERSION`、README 徽章和插件清单写的是同一个号码；
- 更新合同检查机：如果内容变了但插件版本没变，就阻止发布，避免用户收不到更新；
- 发版准备机：一次修改所有需要同步的版本号。

`.github/workflows/` 会在提交和发版时自动运行这些检查。它解决的是“人会忘记，机器不会”。

## DBS 也不是每一处都值得抄

深度调研后发现几处真实问题：

1. 29 个 Skill 已经带来路由重叠和维护成本，新用户更不应该背名字。
2. 大部分 Skill 仍然主要靠 `SKILL.md`，只有少数重任务有脚本和验收。
3. 调研时知识来源图仍写“26 Skills”，主 README 已变成 28 个功能 Skill 加更新入口，说明图会过期。
4. 原子库个别旧卡片仍引用已经变化的 Skill 名称，知识库也需要持续清理。
5. Release 自动说明偏工程化，用户不一定一眼知道新版具体解决了什么。
6. DBS 没有 TC 当前已有的 Codex 插件清单和每个 Skill 的 `agents/openai.yaml`，在 Codex 展示元数据上 TC 反而更完整。

所以我们的原则是：学习 DBS 的分层和闭环，不用“文件夹数量”冒充能力。

## TC 现在到底差在哪里

| 方面 | TC 当前状态 | 要补什么 |
| --- | --- | --- |
| 统一入口 | 已经有 `/tc`，但路由规则分散在长文中 | 明确新任务、继续任务、保存/恢复三种入口状态 |
| 专项 Skill | 有诊断、文案、行动、更新 | 新增一个真正有脚本的 `tc-state`，不拆成三个空 Skill |
| 长期状态 | 几乎没有 | 本地保存、恢复、列表和阶段报告 |
| 知识库 | 有规则和模板，真实内容很少 | 先放少量来自公开推文、带日期与链接的知识原子 |
| 脚本 | 只有检查、构建、发版 | 增加给用户任务直接使用的状态脚本 |
| 验收 | 有文字场景 | 增加状态脚本的自动测试和知识原子结构检查 |
| 说明与发布 | 已经比较完整 | 用实际新增能力写 Release，不再只说“优化体验” |

## 这次迭代的取舍

这次不把 TC 从 5 个 Skill 暴涨到 29 个。只做四件能被检查的事：

1. 新增 `tc-state`：保存、恢复和汇总创业判断；
2. 给 `tc-state` 配可执行脚本，而不是只写提示词；
3. 建立第一批有公开来源的 TC 知识原子，并清楚标记为历史表达；
4. 把 `/tc` 改成更明确的单步路由入口。

判断这次更新有没有用，不看目录数量，看三个证据：用户下次能不能接着上次继续；重要结论能不能追溯来源；构建和自动检查能不能发现明显错误。
